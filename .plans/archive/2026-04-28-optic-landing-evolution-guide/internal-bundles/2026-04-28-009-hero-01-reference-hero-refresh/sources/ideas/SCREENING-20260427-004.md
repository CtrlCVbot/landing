## 2. 판정

**Go**. 이 개선은 단순 색상 보정이 아니라 Hero의 visual target을 다시 고정하는 작업이다. `hero-01` 레퍼런스를 기준으로 full-bleed liquid gradient, 중앙 집중형 타이포그래피, pointer-reactive field, light/dark palette를 재정의하면 현재 구현과 사용자 기대 사이의 gap을 직접 줄일 수 있다.

다만 바로 구현으로 들어가면 또다시 "비슷하지만 원하는 방향은 아닌" 결과가 나올 수 있다. 다음 단계에서는 구현 후보를 먼저 비교하고, 레퍼런스 충실도와 현재 landing 전환 목적 사이의 우선순위를 acceptance criteria로 고정해야 한다.

Standard로 잡는 이유는 코드량이 많아서가 아니라, 첫 화면과 theme system에 닿는 visual risk가 크기 때문이다. MVP로 구현하더라도 draft에서 visual target과 구현 후보를 먼저 고정하는 것이 맞다.

## 4. 주요 결정 필요 사항

| 항목 | 결정 방향 |
|---|---|
| 구현 방식 | `CSS-only`, `Canvas 2D`, `WebGL/Three.js` 후보를 draft에서 비교 |
| 레퍼런스 충실도 | `hero-01`의 full-bleed field, 중앙 집중형 hierarchy, 색 field 밀도를 우선 기준으로 둔다. |
| 색상 조합 | reference palette 직접 복사가 아니라 현재 light/dark theme token 파생값으로 정의 |
| Hero 구성 | CTA와 `DashboardPreview`는 유지하되, first viewport에서 gradient와 headline의 인상이 묻히지 않게 재배치 검토 |
| Controls UI | color scheme buttons, color adjuster, custom cursor는 기본 제외 |
| 접근성 | `prefers-reduced-motion`, text contrast, pointer event 차단 여부를 필수 검증 |

## 5. Acceptance Criteria 후보

- Hero 첫 화면은 `hero-01`처럼 full-bleed liquid gradient가 주 시각 신호로 보인다.
- light mode와 dark mode 모두 기존 landing theme과 조화된다.
- headline과 CTA는 배경 위에서도 충분한 대비와 가독성을 가진다.
- CTA와 주요 링크 클릭 영역은 background layer에 가로막히지 않는다.
- 모바일에서 가로 overflow가 없고 text/button/`DashboardPreview`가 서로 겹치지 않는다.
- `prefers-reduced-motion` 환경에서는 움직임이 꺼지거나 매우 약해진다.
- `hero-01`의 controls UI는 제품 UI로 노출하지 않는다.
- 새 dependency가 필요하면 bundle size, 성능, fallback 대안을 함께 보고한다.
