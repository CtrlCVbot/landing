## 2. 판정

**Go 제안**. 이 아이디어는 기존 `dash-preview`를 새 기능으로 확장하기보다, 같은 시나리오를 더 잘 보이게 만드는 표현 개선이다. AI 입력창, 추출 버튼, 추출 결과, 적용 대상 폼의 연결을 단계별로 강조하면 방문자가 흐름을 더 빠르게 이해할 수 있다.

다만 바로 구현으로 들어가기 전에는 포커스 대상과 좌표 모델을 먼저 문서화해야 한다. 확대·이동 효과는 시각적으로 강하기 때문에, 과하면 제품 데모보다 장식처럼 보일 수 있다.

## 3. Lane 판정

**Lite**.

| Trigger | 해당 | 근거 |
|---|:---:|---|
| 3개 이상 화면 변경 | 아니오 | Hero 내부 `dash-preview` 한 영역의 표현 개선이다. |
| DB 스키마 변경 | 아니오 | 데이터 저장 구조 변경 없음 |
| 외부 API 연동 | 아니오 | mock/demo animation only |
| 보안/인증 흐름 변경 | 아니오 | 사용자 인증이나 권한 흐름과 무관 |
| 2개 이상 도메인 영향 | 아니오 | landing demo presentation 범위 |
| 1주 이상 구현 | 경계 | 좌표 모델과 browser QA를 포함하면 늘어날 수 있으나, 기능 범위 자체는 국소적이다. |

검증은 Lite 범위로 시작하되, 시각 회귀 위험이 있으므로 browser screenshot 확인과 `prefers-reduced-motion` 확인을 필수로 둔다.

## 4. 주요 결정 필요 사항

| 항목 | 결정 방향 |
|---|---|
| 포커스 방식 | 전체 캔버스를 이동·확대할지, 단계별 장면을 별도 레이어로 구성할지 `/plan-draft`에서 결정 |
| 단계 수 | 입력창, 추출 버튼, 추출 결과, 적용 대상 폼, 완료 상태를 최소 기준으로 정리 |
| 모바일 처리 | 전체 UI 축소보다 장면 중심 전환으로 단순화하는 방향 검토 |
| 접근성 | `prefers-reduced-motion`에서는 scale/translate 이동 폭을 줄이고 opacity 또는 highlight 중심으로 대체 |
| 기존 인터랙티브 모드 | 자동 재생 step data와 공유 가능한지 확인 |

## 5. Acceptance Criteria 후보

- 단계별 포커스 대상이 데이터로 정의된다.
- 카톡 입력 단계에서는 AI 입력창이 화면 중심에서 확대되어 보인다.
- 추출하기 단계 이후에는 추출 결과 영역으로 자연스럽게 이동한다.
- 추출정보 적용 단계에서는 관련 입력 폼과 채워지는 값이 함께 보인다.
- 자동 재생과 사용자 step 이동이 같은 단계 정의를 공유한다.
- 모바일에서는 텍스트와 핵심 UI가 잘리지 않는다.
- `prefers-reduced-motion` 환경에서 과한 줌 이동이 줄어든다.
