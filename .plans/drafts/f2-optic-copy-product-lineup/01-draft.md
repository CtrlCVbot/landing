## 2. 사용자 스토리

1. **As a** 랜딩 방문자, **I want** `OPTIC`의 제품 라인업을 역할별로 이해하기를, **so that** 내 업무에 맞는 제품을 빠르게 찾을 수 있다.
2. **As a** 주선사 운영자, **I want** 기능 설명이 `오더`, `운송`, `정산`, `세금계산서` 흐름으로 이어지기를, **so that** 단순 기능 목록이 아니라 실제 업무 흐름으로 받아들일 수 있다.
3. **As a** 화주 또는 의사결정자, **I want** OPTIC이 화주와 주선사의 운영 방식에 맞춰 조율되고 커스텀될 수 있음을 바로 이해하기를, **so that** 우리 조직의 업무 방식에 적용 가능한 서비스로 판단할 수 있다.
4. **As a** 화주 또는 의사결정자, **I want** 외부 브랜드명 나열보다 줄어드는 수작업과 누락 방지를 먼저 보기를, **so that** 도입 가치가 더 명확해진다.
5. **As a** 이후 F3를 구현할 개발자, **I want** 기존 섹션 카피 기준이 먼저 정리되기를, **so that** 업무 매뉴얼형 섹션을 만들 때 같은 설명을 반복하지 않는다.

## 3. 러프 요구사항

### 3.1 카피 기준

- **R1**: 고객 화면의 주 브랜드는 `OPTIC`으로 유지한다.
- **R2**: `Optic Cargo`, `서비스 테스트`, `테스트 서버`, `데모 테스트` 같은 이전/내부 검증성 문구를 랜딩 본문에서 제거한다.
- **R3**: `Cargo` 중심 표현은 필요한 맥락을 제외하고 `오더`, `운송`, `운영 조율`, `정산` 중심 표현으로 바꾼다.
- **R4**: `OPTICS`는 footer/About의 `Powered by OPTICS` 또는 내부 엔진 보조 설명 수준으로 제한한다.
- **R5**: `화물맨`은 외부 플랫폼 고유명으로 유지하고, 그 외 외부 AI/알림/전자세금계산서 서비스명은 일반 기능명으로 처리한다.
- **R6**: `OPTIC`의 `Coordination` 의미를 고객 가치로 풀어, 화주와 주선사별 운영 방식·정산 기준·연동 흐름에 맞춰 커스텀 가능한 서비스임을 주요 메시지로 반영한다.

### 3.2 섹션별 범위

- **R7**: `src/lib/constants.ts`에서 기능 카드, 제품 라인업, 연동 문구의 기준 데이터를 정리한다.
- **R8**: `src/components/sections/features.tsx`는 `주문 관리` 중심 표현을 `AI 오더 등록`, 정산, 전자세금계산서 흐름과 연결한다.
- **R9**: `src/components/sections/problems.tsx`는 브랜드명 나열보다 수작업 감소, 전송 누락 방지, 정산 누락 방지를 먼저 말한다.
- **R10**: `src/components/sections/products.tsx`는 현재 구현 제품을 `주선사용 운송 운영 콘솔`과 `화주용 운송 요청 포털`로 먼저 설명하고, `OPTIC Broker`, `OPTIC Shipper`는 보조 라벨로 병기한다. `OPTIC Carrier`, `OPTIC Ops`, `OPTIC Billing`은 구현 예정으로 분리한다.
- **R11**: `src/components/sections/integrations.tsx`는 `화물맨`, `AI 오더 등록`, `전자세금계산서`, `주소/거리 계산`처럼 기능명 중심으로 설명한다.
- **R12**: `src/components/sections/hero.tsx`는 구조 변경 없이 CTA 문구 스캔 대상에만 포함하되, PRD에서 hero 보조 문구에 커스텀 가능성 메시지를 넣을지 검토한다.

### 3.3 Out-of-scope

## 5. OPTIC 의미 기반 카피 체계

`OPTIC`의 의미는 랜딩에서 약어 설명으로 길게 노출하기보다, 고객이 바로 이해할 수 있는 가치 문장으로 풀어쓴다. 핵심은 **화주와 주선사마다 다른 운송 운영 방식을 하나의 흐름으로 최적화하고 조율한다**는 메시지다.

### 5.1 의미별 카피 역할

| OPTIC 의미 | 고객 가치로 풀기 | 카피 방향 |
|---|---|---|
| Optimized | 회사별 운영 기준에 맞게 최적화 | `회사마다 다른 운송 운영을 우리 방식에 맞게 정리` |
| Platform | 오더, 배차, 연동, 정산을 한 흐름으로 연결 | `오더부터 정산까지 하나의 운영 흐름` |
| Transport | 운송 실무 전체를 대상으로 함 | `운송 요청, 배차, 정산, 증빙까지` |
| Intelligence | AI가 오더를 읽고 누락을 줄임 | `AI가 반복 입력과 누락을 줄임` |
| Coordination | 화주·주선사별 방식에 맞춰 조율 | `각 조직의 업무 방식에 맞춰 조율되는 플랫폼` |

### 5.2 핵심 카피 후보

| 용도 | 카피 후보 |
|---|---|
| 메인 메시지 | 화주와 주선사마다 다른 운송 운영, OPTIC이 맞춰 정리합니다. |
| 보조 메시지 | 오더부터 정산까지, 우리 방식에 맞게 조율되는 운송 운영 플랫폼. |
| AI 연결 메시지 | AI 오더 등록에서 정산까지, OPTIC은 운영 방식에 맞춰집니다. |
| 운영 가치 메시지 | 반복 입력은 줄이고, 회사별 운송 흐름은 더 선명하게. |
| 역할 연결 메시지 | 화주·주선사별 프로세스를 하나의 운영 흐름으로 연결합니다. |

### 5.3 제품 라인업 카피 후보

| 표시 제목 | 보조 라벨 | 상태 | 추천 카피 |
|---|---|---|---|
| 주선사용 운송 운영 콘솔 | `OPTIC Broker` | 구현 대상 | 오더 접수, 배차, 화물맨 연동, 정산 흐름을 주선사 업무 방식에 맞춰 관리합니다. |
| 화주용 운송 요청 포털 | `OPTIC Shipper` | 구현 대상 | 화주별 요청 양식과 진행 확인 방식을 맞춰, 운송 의뢰부터 상태 확인까지 한 흐름으로 정리합니다. |
| 운송사용 배차 수행 도구 | `OPTIC Carrier` | 구현 예정 | 운송사와 차주가 필요한 배차 수행 정보를 간결하게 확인합니다. |
| 운영 조율 콘솔 | `OPTIC Ops` | 구현 예정 | 여러 고객사와 운영 기준을 한 화면에서 조율합니다. |
| 정산 관리 도구 | `OPTIC Billing` | 구현 예정 | 회사별 정산 기준과 세금계산서 흐름을 놓치지 않게 관리합니다. |

### 5.4 PRD에 넣을 기준 문장

> OPTIC은 하나의 고정된 운송 관리 화면이 아니라, 화주와 주선사마다 다른 오더 접수 방식, 연동 방식, 정산 기준을 하나의 운영 흐름으로 조율하는 맞춤형 운송 운영 플랫폼이다.

## 6. 매뉴얼형 섹션 제안

F2에서 실제 신규 섹션을 구현하지는 않지만, 기존 섹션 카피와 F3 업무 매뉴얼형 섹션이 같은 흐름을 바라보도록 아래 구조를 PRD 입력으로 남긴다.

| 단계 | 섹션 역할 | 보여줄 메시지 |
|---|---|---|
| 1. 운영 방식 선택 | 화주형, 주선사형, 정산 중심형 등 업무 기준을 먼저 보여줌 | 시작점부터 회사별 방식에 맞춰진다는 인상 |
| 2. AI 오더 등록 | 회사별 오더 양식이 달라도 AI가 핵심 정보를 정리 | 입력 형식이 달라도 운영 흐름으로 변환 |
| 3. 상하차지 재사용 | 자주 쓰는 장소, 담당자, 메모를 업무 방식에 맞게 축적 | 반복 입력 감소와 현장 정보 재사용 |
| 4. 화물맨 연동 | 한 번 입력한 운송 정보를 외부 채널로 연결 | 중복 등록과 전송 누락 감소 |
| 5. 정산 기준 적용 | 화주별 청구, 주선사별 매출/매입 정산 기준 반영 | 회사별 계산 기준을 흐름 안에서 관리 |
| 6. 세금계산서 확인 | 정산 이후 증빙 상태까지 이어서 관리 | 정산 완료 이후 후속 업무 누락 방지 |

매뉴얼형 섹션은 각 단계마다 `무엇을 입력하는가`, `무엇이 자동으로 이어지는가`, `무엇을 회사별로 맞출 수 있는가`를 짧게 보여주는 구조를 우선한다.

## 8. Acceptance Criteria

- 랜딩 코드와 테스트 대상 문구에서 `Optic Cargo`가 남지 않는다.
- 고객 CTA나 본문에 `서비스 테스트`, `테스트 서버`, `데모 테스트`가 남지 않는다.
- 제품 라인업은 현재 구현 대상 `주선사용 운송 운영 콘솔`, `화주용 운송 요청 포털`이 우선 보이고, `OPTIC Broker`, `OPTIC Shipper`는 보조 라벨로 병기된다.
- `OPTIC Carrier`, `OPTIC Ops`, `OPTIC Billing`은 구현 예정으로만 구분된다.
- 화주/주선사별 운영 방식에 맞춘 커스텀 가능성이 OPTIC의 핵심 가치로 보인다.
- `OPTIC`의 의미가 약어 설명이 아니라 최적화, 플랫폼, 운송, AI, 조율의 고객 가치 문장으로 풀린다.
- 매뉴얼형 섹션으로 확장 가능한 단계 흐름이 PRD에 남는다.
- `화물맨` 외 외부 브랜드명은 고객 화면에서 일반 기능명으로 대체된다.
- 여러 섹션의 긴 한글 문구가 desktop/mobile에서 겹치지 않는다.
- F3가 이어받을 수 있도록 기존 섹션과 신규 업무 흐름의 역할 구분이 PRD에 남는다.

| 제품명/역할명 스캔 | `주선사용 운송 운영 콘솔`, `화주용 운송 요청 포털`이 1차 제목으로 보이고 `OPTIC Broker`, `OPTIC Shipper`는 보조 라벨로 노출되는지 확인 |
| 커스텀 메시지 스캔 | 화주/주선사별 맞춤, 커스텀, 운영 방식 조율 메시지가 주요 섹션에 반영됐는지 확인 |
| OPTIC 의미 스캔 | `Optimized`, `Platform`, `Transport`, `Intelligence`, `Coordination`을 직접 나열하기보다 고객 가치 문장으로 풀었는지 확인 |
| 매뉴얼형 흐름 확인 | 운영 방식 선택 → AI 오더 등록 → 상하차지 → 화물맨 → 정산 → 세금계산서 흐름이 PRD에 반영됐는지 확인 |
| 외부 브랜드명 스캔 | `화물맨` 외 특정 외부 브랜드명 잔존 여부 확인 |
| 테스트 | 관련 섹션 렌더 테스트 또는 constants 테스트 추가/갱신 |
| 타입 체크 | TypeScript 오류 없음 |
| lint | JSX/문구 변경 중 lint 오류 없음 |
| 수동 확인 | desktop/mobile에서 섹션 카드와 버튼 텍스트 겹침 없음 |

## 10. 리스크와 완화

| 리스크 | 수준 | 완화 |

| OPTIC 의미 설명이 내부 브랜딩처럼 길어질 수 있다 | low | 약어 자체보다 고객 가치 문장과 매뉴얼형 흐름에 우선 반영한다 |

| 긴 한글 문구가 카드 UI를 깨뜨릴 수 있다 | medium | 구현 후 desktop/mobile spot check를 검증 게이트로 둔다 |
