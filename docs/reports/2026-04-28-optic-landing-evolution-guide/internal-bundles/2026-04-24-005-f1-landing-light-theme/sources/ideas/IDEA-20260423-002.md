## 1. 원문 요약

dash-preview Phase 3 가 2026-04-22 archived 상태로 종료된 직후, 사용자가 실제 미리보기를 사용해 본 뒤 전달한 10 건 피드백 중 **이슈 [1]** 을 단일 Feature 로 분리해 처리한다. 원천: [`.plans/archive/dash-preview-phase3/improvements/issues.md`](../../archive/dash-preview-phase3/improvements/issues.md). 본 Feature 는 **landing 사이트 전역** (dash-preview 는 서브셋) 을 대상으로 라이트 모드 전환 인프라를 구축한다 (2026-04-23 범위 확장 확정).

- **이슈 [1]**: 라이트/다크 모드 부재 — landing 사이트 전역 (hero · features · pricing · footer · navbar 등 메인 섹션 + dash-preview 하위 컴포넌트 + shared UI) 이 `bg-white/5`·`text-white`·`from-gray-900/50`·`border-gray-800` 등 **다크 팔레트 전용** 하드코딩 클래스로 구성되어, 사용자의 라이트 모드 선호 / 주간 환경 / 밝은 디스플레이 컨텍스트를 수용하지 못한다. `prefers-color-scheme: light` 환경에서 그대로 다크 팔레트가 노출되어 대비가 과도하고 접근성 (WCAG AA) 가 흔들린다. Phase 3 archive 직후 사용자 피드백: "라이트 모드가 없어서 사용하기 어렵다" (2026-04-22).

## 2. 배경 (Why)

landing 사이트는 다크 팔레트 단일 기준으로 구축되어 다음 한계가 있다:

1. **접근성 확장 차단**: `prefers-color-scheme: light` 사용자에게 대비 과도 + WCAG AA 흔들림. axe-core 라이트 모드 검증 수행 불가 (landing 전역).
2. **사용자 선호 대응 부재**: 주간 환경·밝은 디스플레이 컨텍스트에서 **전체 페이지** 가독성 저하.
3. **차기 Epic 의 디자인 시스템 부채**: 다크 전용 하드코딩 클래스가 landing 전반에 쌓여 있어, 향후 토큰 기반 디자인 시스템 도입 시 광범위 마이그레이션 비용 발생.

본 Feature 는 Epic §2 성공 지표 4 (라이트/다크 양 팔레트 지원 + 전환 경로) 의 전담 Feature 로, **landing 사이트 전역** 을 대상으로 토큰 이중화 + ThemeProvider + 컴포넌트 스윕을 통합 수행해 Phase A 내 MVP 달성을 목표한다. dash-preview 는 본 Feature 의 **대상 서브셋** 이며 별도 처리 없이 동일 토큰 치환 규칙이 적용된다.

## 3. 범위 (Scope)

### A. 토큰 이중화 (전역 CSS 팔레트 이중화)

- **`src/app/globals.css`**: `:root` (라이트) + `[data-theme="dark"]` (또는 `.dark`) 의 CSS 변수 2 세트 정의
  - 변수 예: `--background`, `--foreground`, `--muted`, `--accent`, `--border`, `--card`, `--card-foreground`, `--primary`, `--secondary`, `--destructive` 등
- **`tailwind.config.ts`**: `darkMode: ['class', '[data-theme="dark"]']` + CSS 변수 기반 컬러 매핑 (`colors: { background: 'var(--background)', ... }`)

### B. ThemeProvider 도입

- **`src/app/layout.tsx`**: `ThemeProvider` 또는 `data-theme` 속성 주입 로직
- **결정 포인트 (`/plan-draft` 확정)**:
  - `next-themes` 라이브러리 사용 vs 직접 구현
  - 초기 모드: `prefers-color-scheme` 시스템 follow / 수동 토글 / 강제 라이트 (기본) 중 선택
  - SSR hydration mismatch 대응 전략

### C. landing 전역 컴포넌트 스윕 (다크 하드코딩 클래스 토큰 치환)

**landing 사이트의 모든 컴포넌트** (`src/app/**/page.tsx`, `src/app/**/layout.tsx`, `src/components/**/*.tsx`) 를 대상으로 다크 전용 클래스를 토큰 기반 클래스로 교체.

**주요 스윕 영역** (`/plan-draft` 단계에서 전수 조사 + 확정):

1. **1차 — landing 메인 섹션**: `src/app/page.tsx` · `src/app/layout.tsx` · hero · features · pricing · testimonials · footer · navbar 등 상위 구성 요소
2. **2차 — dash-preview 서브셋** (7 개 파일):
   - `datetime-card.tsx` · `estimate-info-card.tsx` · `settlement-section.tsx` · `transport-option-card.tsx` · `order-form/index.tsx` · `preview-chrome.tsx` · `interactive-tooltip.tsx`
3. **3차 — 공통 UI**: `src/components/ui/` (shadcn 등) · 기타 shared 컴포넌트

**치환 대상 예시**:
- `bg-white/5` → `bg-card` (또는 `bg-muted/50`)
- `text-white` → `text-foreground`
- `from-gray-900/50` → 토큰 그라데이션 (`from-background/50` 등)
- `border-gray-800` → `border-border`
- `bg-gray-*` / `text-gray-*` / `border-gray-*` / `bg-black` / `bg-slate-*` 전반 토큰 치환

**전수 조사 방법** (`/plan-draft` 확정):
- `grep -rn "bg-white\|text-white\|from-gray\|to-gray\|border-gray\|bg-black\|bg-slate" src/` 기반 후보 목록 생성
- 각 후보 라인마다 **토큰 매핑 결정표** 작성 (토큰 이중화 §A 결과와 정합)
- 섹션별 PR 분할 (hero / features / pricing / footer / dash-preview 등 5~6 개 PR) 로 merge 리스크 분산

### D. 전환 트리거 UI (선택)

- 토글 버튼 배치 위치 결정 (`navbar` 우측 상단 / footer / 전면 자동 전환 중)
- 아이콘 선택 (sun/moon 등)
- 결정 시점: `/plan-draft` 단계

### Out-of-scope

- 라이트 모드를 계기로 한 **대규모 디자인 시스템 리팩토링** (차기 Epic 후보, Epic §3 Out-of-scope 와 정합) — 토큰 치환에 한정하며 토큰 값·스케일 재설계는 포함하지 않음
- landing **이외 영역** 의 라이트 모드 대응 (별도 앱·관리자·대시보드·docs 등은 본 Feature 범위 밖)
- 다국어(i18n) 연동 (Epic §3 Out-of-scope 와 정합)
- 모바일 전용 팔레트 분리 (Phase 5+ 후보)
- 브랜드 컬러·로고 등 **팔레트 외 자산** 의 라이트/다크 분기 (별도 Theme 후보)

### 9-2. 결정 포인트 6 건 확정

1. **ThemeProvider 구현 방식**: **next-themes 채택 (확정)**. Next.js + React 18 App Router 사실상 표준 (vercel·shadcn 공식). gzip ~1.5kB. `<script>` 인라인 주입으로 FOUC 차단 + `data-theme` 서버/클라이언트 동기 설정 — 직접 구현 3~4 인·일 대비 1 인·일 이내 해결. Screening §8-4 리스크 "SSR hydration" 직접 완화.
2. **초기 모드**: **`prefers-color-scheme` 시스템 follow + 수동 토글 병행 (확정)**. `defaultTheme="system"` + `enableSystem`. WCAG 2.2 + Apple HIG · Material 공통 원칙 (시스템 설정 = 사용자 접근성 선택 1차 signal). IDEA §4-2 "사용자 선호 대응" 근본 실현. 기존 다크 환경 방문자 이탈 리스크 없음.
3. **SSR hydration mismatch 대응**: **3 중 방어 조합 (확정)**. (1) `<html suppressHydrationWarning>`, (2) `<ThemeProvider disableTransitionOnChange>`, (3) `ThemeToggle` 컴포넌트 `mounted` state. 각 방어가 역할 분담 — 경고 억제 · 시각 flash 억제 · 아이콘 자체 mismatch 방어. shadcn 공식 theming 가이드 표준 조합.
4. **전환 트리거 UI 배치**: **navbar 우측 상단 (확정)**. 스크롤 위치 무관 지속 노출 (특히 sticky) → 발견성 최고. GitHub · Vercel · Stripe · Linear 랜딩 공통 관행 → 사용자 학습 곡선 0. footer (발견성 부족) · 전면 자동 전환 (접근성 위반 + `prefers-reduced-motion` 위반) 거절.
