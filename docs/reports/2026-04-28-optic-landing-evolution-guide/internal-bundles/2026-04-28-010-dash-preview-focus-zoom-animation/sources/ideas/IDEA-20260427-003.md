## 1. 아이디어 요약

`dash-preview`의 자동 재생 데모를 고정된 축소 화면 중심에서, 단계별로 중요한 UI 파트만 확대해서 보여주는 포커스 줌 애니메이션으로 개선한다.

현재 단계와 관련된 영역만 화면 중심에 `zoom in/out` 또는 `scale up`으로 크게 보여주고, 다음 단계로 넘어갈 때는 관련 컴포넌트 위치로 자연스럽게 이동한다. 예를 들어 카톡 메시지를 입력하는 단계에서는 AI 입력창만 확대하고, 추출하기 버튼을 누른 뒤에는 바로 아래 추출 결과 영역으로 화면이 이동한다.

## 2. 배경

기존 `dash-preview`는 AI 화물 등록 흐름을 시네마틱 축소 뷰로 보여준다. 다만 전체 UI가 한 프레임 안에 계속 보이면, 방문자가 각 단계에서 어디를 봐야 하는지 바로 이해하기 어렵다.

이 개선은 제품 화면 전체를 보여주는 목적보다, "지금 어떤 일이 일어나고 있는지"를 단계마다 또렷하게 전달하는 데 초점을 둔다.

## 3. 기대 효과

- 방문자가 AI 입력, 추출, 적용, 정보 입력 흐름을 더 쉽게 따라간다.
- 작은 축소 화면에서도 핵심 파트가 크게 보여서 모바일과 좁은 화면에서 이해도가 올라간다.
- Hero 데모가 단순 미리보기보다 제품 사용 흐름을 안내하는 짧은 시연처럼 보인다.
- 추출 버튼, 추출 결과, 입력 폼처럼 서로 이어지는 컴포넌트 관계가 더 명확해진다.

## 4. 원하는 단계 흐름

| 단계 | 포커스 영역 | 화면 전환 의도 |
|---|---|---|
| 1 | 전체 AI 등록 화면 | 제품 화면의 전체 구조를 짧게 보여준다. |
| 2 | 카톡 텍스트 입력창 | 입력창이 확대되고 메시지 입력 애니메이션이 진행된다. |
| 3 | 추출하기 버튼 | 버튼 클릭 동작이 강조되고 다음 영역으로 이동할 준비를 보여준다. |
| 4 | 추출 결과 영역 | 버튼 아래 생성된 추출 결과 컴포넌트로 화면이 이동한다. |
| 5 | 추출정보 적용 대상 폼 | 관련 입력 폼 영역으로 이동하고 데이터가 채워지는 장면을 보여준다. |
| 6 | 완료 상태 | 전체 화면 또는 핵심 요약 영역으로 다시 돌아와 결과를 보여준다. |

## 5. 구현 검토 포인트

- 단계별 포커스 좌표를 데이터로 관리할 수 있는지 확인한다.
- `transform: scale()`, `translate`, `clip-path`, `overflow: hidden` 같은 CSS 기반 방식으로 충분한지 검토한다.
- 고정 사각 프레임 안에 전체 UI를 계속 노출할지, 현재 단계 외 영역은 프레임 밖으로 밀어낼지 결정한다.
- 자동 재생과 사용자 클릭 모드가 같은 단계 데이터를 공유할 수 있는지 확인한다.
- 모바일에서는 전체 축소 뷰 대신 단계별 장면 전환 중심으로 단순화할지 검토한다.

## 6. Acceptance Criteria 초안

- 각 단계는 하나의 주요 포커스 영역을 가진다.
- 포커스 영역은 단계 전환 시 자연스럽게 확대, 축소, 이동한다.
- 카톡 입력 단계에서는 AI 입력창이 화면 중심에서 충분히 크게 보인다.
- 추출하기 단계 이후에는 추출 결과 영역으로 시선이 이동한다.
- 추출정보 적용 단계에서는 관련 입력 폼과 채워지는 데이터가 함께 보인다.
- 자동 재생 흐름이 기존 `dash-preview` 핵심 시나리오를 깨지 않는다.
- `prefers-reduced-motion` 환경에서는 과한 줌 이동을 줄인 대체 표현을 제공한다.
