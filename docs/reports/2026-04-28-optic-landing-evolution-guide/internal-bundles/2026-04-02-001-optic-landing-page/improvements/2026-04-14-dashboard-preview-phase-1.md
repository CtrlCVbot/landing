## 1. 문서 목적

이 문서는 `apps/landing`의 Hero 영역에 들어갈 `Dashboard Preview`의 1차 기획안을 정리한 문서다.

이번 문서의 목적은 다음 3가지다.

1. 정적인 이미지 또는 영상 대신 사용할 인터랙티브 프론트 데모의 방향을 정의한다.
2. 실제 운송 주문 등록 화면의 어떤 요소를 유지하고, 어떤 요소를 랜딩용으로 단순화할지 정리한다.
3. 이후 UI 설계, 컴포넌트 구현, mock data 설계 논의로 바로 이어질 수 있는 기준 문서를 만든다.

이번 단계는 구현 문서가 아니라 기획 문서다.  
즉, 실제 컴포넌트 구조나 코드 레벨 상세보다는 사용자에게 어떤 인상을 줄지, 어떤 흐름을 보여줄지, 어디까지 보여주고 어디까지 생략할지를 우선 정의한다.

---

## 2. 배경

현재 랜딩 페이지 Hero에는 `Dashboard Preview` placeholder가 존재하지만, 제품의 실제 사용감을 전달하기에는 정보량과 설득력이 부족하다.

운송 운영 SaaS는 일반적인 소비자 서비스와 달리 다음 요소가 중요하다.

- 업무 흐름이 어떻게 이어지는지
- 입력값이 어떻게 정리되는지
- 복잡한 조건이 어떻게 화면에서 구조화되는지
- 사용자가 실제로 다룰 법한 정보 밀도와 톤이 있는지

이 때문에 단순한 정적 이미지 한 장만으로는 제품 가치가 충분히 전달되지 않을 수 있다.

영상도 대안이 될 수 있지만, 이번 랜딩에서는 다음 이유로 UI 기반 프리뷰가 더 적합하다.

- 텍스트와 숫자가 더 또렷하게 보인다.
- 반응형과 유지보수 대응이 쉽다.
- 랜딩 전체 톤앤매너와 자연스럽게 통합된다.
- hover, step 이동, 상태 강조 같은 가벼운 상호작용을 추가할 수 있다.
- 실제 제품 화면처럼 보이는 신뢰감을 줄 수 있다.

이번 Dashboard Preview는 `실제 기능을 수행하는 화면`이 아니라 `실제 기능처럼 보이는 연출형 프론트 데모`를 목표로 한다.

## 3. 문제 정의

현재 Hero 영역의 핵심 문제는 다음과 같다.

| 항목 | 현재 상태 | 문제 |
|---|---|---|
| 제품 이해 | 텍스트 중심 | 실제로 무엇을 하는 제품인지 한눈에 와닿지 않을 수 있음 |
| 시각 설득력 | placeholder 수준 | 랜딩의 핵심 비주얼로서 존재감이 약함 |
| 업무 신뢰감 | 부족 | 운송 운영 SaaS 특유의 현실감이 충분히 드러나지 않음 |
| 기능 전달 방식 | 설명 위주 | 입력, 정리, 요약 흐름이 시각적으로 보이지 않음 |

정리하면, 현재 Hero는 “무슨 제품인가”는 설명하지만 “어떻게 작동하고 어떤 느낌인가”는 전달하지 못하는 상태다.

## 4. 기획 목표

### 4-1. 핵심 목표

- Hero에서 제품의 첫인상을 강화한다.
- 운송 주문 등록 workflow를 가장 직관적인 방식으로 압축해서 보여준다.
- 사용자가 랜딩에 들어오자마자 “입력은 복잡해 보여도 결과는 빠르게 정리되는 제품”이라는 인상을 받게 한다.

### 4-2. 사용자에게 남기고 싶은 인상

- 실제 현업용 SaaS 화면 같다.
- 업무형 제품인데 복잡하게 느껴지지 않는다.
- 입력과 요약이 잘 연결되어 있다.
- 영상보다 더 믿음 가고, 이미지보다 더 살아 있다.

### 4-3. 성공 기준

- Hero 첫 화면만 보고도 제품 카테고리와 가치가 대략 이해된다.
- Preview가 CTA를 방해하지 않고 오히려 관심을 증폭시킨다.
- 모바일에서도 “작은 축소판”이 아니라 “의미 있는 핵심 장면”으로 전달된다.
- 구현 이후에도 카피, mock data, step 구성이 쉽게 수정 가능해야 한다.

## 5. 섹션 역할 정의

이번 Dashboard Preview는 단순 장식 요소가 아니다. 랜딩 내에서 아래 역할을 담당한다.

### 5-1. 제품 설명 역할

텍스트만으로 설명하기 어려운 제품 경험을 시각적으로 압축해서 보여준다.

### 5-2. 신뢰 형성 역할

업무용 화면 구조, utility copy, 요약 패널, 상태 변화 등을 통해 실제 제품 같은 완성도를 전달한다.

### 5-3. CTA 보조 역할

Preview를 본 사용자가 “도입 문의” 또는 “데모 보기”로 자연스럽게 넘어가게 한다.

### 5-4. 이후 섹션 연결 역할

Hero Preview에서 느낀 인상을 `Problems`, `Features`, `Products` 섹션이 뒤에서 논리적으로 설명하는 구조를 만든다.

## 6. 배치 전략

### 6-1. 권장 배치

기본 배치는 `Hero 메인 비주얼`이다.

- 좌측: headline, short copy, CTA
- 우측: Dashboard Preview window

### 6-2. 이 배치가 적합한 이유

- 제품 메시지와 실제 사용 감각을 첫 화면에서 동시에 보여줄 수 있다.
- 현재 Hero placeholder를 자연스럽게 대체할 수 있다.
- 랜딩 전체에서 가장 강한 시각 앵커를 만들 수 있다.

### 6-3. 기대 효과

- headline만 읽고 지나가는 사용자를 잡을 수 있다.
- Preview만 본 사용자도 제품이 무엇을 돕는지 감을 잡을 수 있다.
- Hero 이후 섹션이 단순 정보 나열이 아니라 “Preview에서 본 장면에 대한 설명”처럼 읽히게 된다.

## 7. 기본 방향과 설계 원칙

### 7-1. 실제 제품 복제가 아니라 랜딩용 재구성

참고 화면은 `.references/code/mm-broker/app/broker/order/register/page.tsx`이지만, 이를 그대로 축소 복제하지 않는다.

유지할 것:

- 주문 등록 흐름의 구조감
- 업무 화면의 톤
- 입력과 요약의 관계
- 옵션과 상태 변화 개념

줄일 것:

- 실제 운영 복잡도
- 예외 처리
- API/DB 연동
- 실제 검색/저장/검증 로직

핵심 원칙은 다음 한 문장으로 정리한다.

> 업무 현실감은 살리고, 운영 복잡성은 제거한다.

### 7-2. 기능 소개가 아니라 workflow 소개

Preview의 핵심은 “무슨 필드가 몇 개 있느냐”가 아니라 “이 업무가 얼마나 빠르게 정리되는가”다.

따라서 사용자가 보는 흐름은 다음 순서를 중심으로 설계한다.

1. 출발/도착 정보가 채워진다.
2. 차량/화물 조건이 정리된다.
3. 옵션이 선택된다.
4. 요약 패널이 즉시 반응한다.
5. 확인 직전 상태까지 정돈된다.

### 7-3. motion은 설명용

애니메이션은 시각적 장식이 아니라 이해를 돕는 도구로만 쓴다.

motion의 역할:

- 시선을 다음 단계로 이동시킨다.
- 바뀐 값을 인지하게 한다.
- “실제로 되는 제품 같다”는 감각을 만든다.

motion이 하면 안 되는 것:

- 정보 전달보다 먼저 눈에 띄는 것
- 화면 자체를 지나치게 복잡하게 만드는 것
- 업무형 제품의 신뢰감을 해치는 과장된 효과

## 8. 화면 컨셉

Dashboard Preview는 “하나의 업무용 앱 창”처럼 보여야 한다.

### 8-1. 전체 구조

| 영역 | 설명 | 목적 |
|---|---|---|
| 상단 chrome | 브라우저/앱 스타일 프레임 | 실제 제품 화면 같은 느낌 제공 |
| 상단 정보 영역 | breadcrumb, 페이지 제목, 보조 설명 | 현재 사용자가 보고 있는 작업 맥락 전달 |
| 좌측 입력 영역 | 주문 등록 핵심 필드 | 입력 흐름을 보여줌 |
| 우측 요약 영역 | 운임, 일정, 옵션, 상태 | 제품 가치 전달의 핵심 |
| 하단 보조 영역 | step indicator 또는 예시 workflow 문구 | 흐름 이해 보조 |

### 8-2. 상단 chrome

포함 요소:

- 작은 상태 점 또는 window controls
- 제품 또는 화면 이름
- 너무 실제 브라우저처럼 디테일하지는 않되 “UI 프레임”이라는 인상을 주는 최소 요소

의도:

- 단순 카드가 아니라 작업 화면처럼 보이게 만든다.

### 8-3. 상단 정보 영역

포함 요소:

- breadcrumb 또는 현재 위치 표시
- `주문 등록` 같은 명확한 작업명
- 짧은 보조 설명

의도:

- 사용자가 지금 무엇을 보고 있는지 즉시 이해하게 만든다.

### 8-4. 좌측 입력 영역

권장 포함 요소:

- 상차지
- 하차지
- 차량 타입
- 중량/톤급
- 화물 종류
- 요청 메모
- 옵션 일부

설계 원칙:

- 실제 화면보다 입력 블록 수를 줄인다.
- 한 번에 모든 필드를 보여주기보다 5~7개 핵심 블록으로 압축한다.
- 각 블록은 “입력 중”처럼 보이되 실제 입력 기능까지는 가지 않는다.

### 8-5. 우측 요약 영역

권장 포함 요소:

- 예상 운임
- 주요 옵션 요약
- 일정 요약
- 차량/화물 요약
- 진행 상태 배지
- 마지막 확인 CTA 또는 완료 직전 상태 표현

의도:

- 사용자는 좌측 폼보다 우측 요약 변화에서 제품의 가치를 더 강하게 느낀다.
- “입력값이 구조화되어 결과로 정리된다”는 메시지를 만든다.

### 8-6. 하단 보조 요소

아래 중 1개만 유지하는 방향을 권장한다.

- step indicator
- 현재 시퀀스 단계명
- `예시 workflow` 같은 짧은 맥락 문구

과한 설명 요소는 랜딩의 집중도를 깨므로 최소화한다.

## 9. 실제처럼 보이게 할 요소 / 생략할 요소

### 9-1. 유지해야 할 요소

- 업무형 레이아웃 구조
- 입력 필드와 요약 패널의 명확한 분리
- 값이 채워지는 순서감
- 숫자 및 상태 배지 변화
- 운송 도메인 특유의 데이터 밀도
- 실무 화면처럼 보이는 spacing과 텍스트 톤

### 9-2. 생략 또는 단순화할 요소

- 실제 API 요청
- DB 연결
- 주소 검색 자동완성
- validation 전체 로직
- copy mode / edit mode / toast / loader
- 운영용 예외 상태
- 사이드바 복잡도
- 상세한 옵션 전체
- 저장 완료 후 라우팅 흐름

### 9-3. 판단 기준

다음 질문에 “아니오”면 과감히 뺀다.

- 이 요소가 제품의 핵심 workflow 이해에 꼭 필요한가?
- 이 요소가 실제 제품 같은 느낌을 만드는 데 기여하는가?
- 이 요소가 랜딩 성능과 가독성을 해치지 않는가?

## 10. mock data 설계 방향

mock data는 단순 filler가 아니라 제품 메시지를 전달하는 장치여야 한다.

### 10-1. 데이터 설계 원칙

- 실제 도메인처럼 보여야 한다.
- 읽기 쉽고 빠르게 이해되어야 한다.
- 값이 바뀔 때 의미 있는 시각 변화가 있어야 한다.

### 10-2. 권장 데이터 그룹

| 그룹 | 예시 |
|---|---|
| 주문 기본 정보 | 주문 번호, 등록 상태 |
| 상차 정보 | 물류센터명, 날짜, 시간 |
| 하차 정보 | 목적지, 날짜, 시간 |
| 차량 정보 | 차량 타입, 중량 |
| 화물 정보 | 화물 종류, 적재 조건 |
| 옵션 정보 | 직송, 지게차, 긴급 등 |
| 정산 정보 | 예상 운임, 운송 옵션 반영 금액 |
| 상태 정보 | 검토 완료, 확인 대기 등 |

### 10-3. 권장 예시 방향

- 상차: 수도권 물류센터
- 하차: 대전 또는 부산
- 차량: 5톤 또는 11톤
- 화물: 팔레트 단위 공산품
- 옵션: 직송, 지게차 상하차, 긴급 배차
- 운임: 옵션에 따라 즉시 숫자가 바뀌는 구조

### 10-4. mock data의 역할

- 화면을 예쁘게 채우는 것
- 도메인을 실제처럼 느끼게 하는 것
- “조건이 반영된다”는 제품 메시지를 보여주는 것

## 11. scripted interaction 설계

이번 Preview의 핵심은 “사용자가 직접 조작하지 않아도 작동하는 제품처럼 느껴지는 것”이다.

### 11-1. 기본 모드

- 기본은 자동 재생
- 사용자 개입은 보조 수단
- hover 또는 step 선택 정도만 허용

### 11-2. 권장 시퀀스

#### Step 1. 시작 상태

- 주문 등록 화면의 기본 골격이 이미 보인다.
- 사용자는 이 화면이 주문 등록용 업무 화면임을 즉시 이해한다.

#### Step 2. 상차 / 하차 정보 입력

- 출발지와 도착지가 순서대로 채워진다.
- 필드 강조와 값 등장만 보여주고, 실제 검색은 하지 않는다.

#### Step 3. 차량 / 화물 조건 선택

- 차량 타입, 톤급, 화물 종류가 정리된다.
- 이 단계에서 업무용 화면의 무게감이 생긴다.

#### Step 4. 옵션 선택 및 반영

- 직송, 지게차 상하차, 긴급 옵션 중 1~2개가 활성화된다.
- 우측 요약 영역의 금액과 상태가 즉시 바뀐다.

#### Step 5. 요약 완료

- 예상 운임, 일정, 옵션, 상태가 정리된 상태로 마무리된다.
- 사용자는 “복잡한 입력이 빠르게 정리된다”는 인상을 받는다.

### 11-3. 수동 상호작용

- hover 시 현재 단계 강조 유지
- step chip 클릭 시 해당 단계로 이동
- 자동 재생 중 개입해도 일정 시간 후 루프로 복귀

### 11-4. 속도 가이드

- 각 단계 2~4초
- 전체 루프 16~22초
- 너무 빠르면 읽히지 않고, 너무 느리면 영상처럼 답답해진다

---

## 12. 대표 시나리오

### 12-1. 시나리오 A: 신규 주문 등록

가장 기본이 되는 대표 시나리오다.

흐름:

1. 상차지 입력
2. 하차지 입력
3. 차량/화물 선택
4. 옵션 선택
5. 우측 요약 정리
6. 확인 직전 상태 도달

전달 메시지:

- 주문 등록이 빠르다.
- 복잡한 정보를 한 화면에서 정리할 수 있다.
- 핵심 값이 자동으로 요약된다.

### 12-2. 시나리오 B: 조건 변경 즉시 반영

MVP 이후 확장용 시나리오다.

흐름:

1. 이미 입력된 주문 정보가 있다.
2. 특정 옵션이 활성화된다.
3. 운임과 상태가 즉시 업데이트된다.
4. 변경된 요약이 강조된다.

전달 메시지:

- 조건 변경에도 판단이 빠르다.
- 운영 중 필요한 즉시성이 있다.

### 12-3. 1차 문서 기준 권장 범위

초기 구현은 시나리오 A 하나만 다루는 것이 적절하다.  
시나리오 B는 확장안으로 남기는 편이 완성도와 일정 측면에서 안전하다.

## 13. 반응형 전략

### 13-1. Desktop

- 좌측 텍스트 / 우측 Preview의 2열 구조
- Preview 내부의 폼과 요약 패널을 모두 보여줌
- step indicator나 하단 보조 설명을 유지할 수 있음

### 13-2. Tablet

- Hero 전체 비율은 유지하되 Preview 내부 정보 밀도를 줄임
- 일부 라벨 또는 보조 설명은 축약
- 요약 패널의 우선순위를 더 높게 둠

### 13-3. Mobile

- 데스크톱 UI 전체 축소판은 지양
- 핵심 단계 1개 또는 2개만 강하게 보여주는 방식이 적합
- 좌우 분할보다 상단 현재 상태 + 하단 요약 중심으로 재배치 권장

### 13-4. 모바일 핵심 원칙

모바일은 “작은 데스크톱”이 아니라 “핵심 메시지만 남긴 축약 데모”여야 한다.

---

## 14. 카피 방향

Preview는 이미 많은 설명을 대신하므로 카피는 짧고 직관적이어야 한다.

### 14-1. Hero 주변 메시지 방향

- 주문 등록부터 정산 전 단계까지, 하나의 흐름으로
- 복잡한 운송 운영을 더 빠르고 명확하게
- 입력은 줄이고, 판단은 더 빠르게

### 14-2. Preview 보조 카피 방향

- 실제 workflow를 기반으로 재구성한 preview
- 주문 조건이 정리되는 흐름을 한눈에
- 빠른 등록과 즉시 반영되는 요약 구조

### 14-3. 톤 원칙

- 감성 문장보다 제품 문장을 우선한다.
- 과장된 marketing copy보다 실제 utility copy에 가깝게 간다.
- 실무형 제품의 신뢰감을 우선한다.

## 15. MVP 정의

### 15-1. MVP 포함

- Hero 내 대형 Dashboard Preview 1개
- 주문 등록 기반 시나리오 1개
- mock data 1세트
- 자동 재생 시퀀스 1개
- hover 또는 step 선택 수준의 가벼운 상호작용
- 모바일 축약 버전
- reduced motion 고려

### 15-2. MVP 제외

- 실제 입력 가능 기능
- DB/API 연동
- 다중 제품 프리뷰 전환
- 복잡한 예외 처리
- 실제 저장/완료 라우팅
- 다중 시나리오

### 15-3. MVP 판단 기준

성공한 MVP의 기준은 “기능이 많아 보이는 것”이 아니라 다음이다.

- 첫 화면에서 메시지가 바로 이해되는가
- 업무형 제품처럼 보이는가
- 랜딩의 시각 중심축 역할을 하는가
- 구현과 유지보수 범위가 통제 가능한가

## 16. 리스크와 대응

| 리스크 | 설명 | 대응 방향 |
|---|---|---|
| 과한 복잡성 | 실제 화면를 너무 많이 가져오면 랜딩 전달력이 낮아짐 | 핵심 필드와 변화만 남김 |
| 과한 연출 | motion이 먼저 보이면 업무 신뢰감이 깨짐 | 강조, 반영, 완료 상태 위주로 제한 |
| 기능 오해 | 사용자가 실제 제품 전체 기능으로 오해할 수 있음 | preview 성격을 은근히 전달 |
| 성능 저하 | Hero는 가장 먼저 보이는 영역이라 부담이 큼 | DOM 단순화, 과한 효과 금지 |
| 모바일 가독성 저하 | 전체 축소판은 읽히지 않음 | 모바일 전용 축약 레이아웃 적용 |

## 17. 이번 문서 기준 결정사항

이번 1차 기획 문서 기준으로 아래 항목은 결정된 상태로 본다.

- Preview는 이미지/영상 대체용 Hero 메인 비주얼이다.
- 주제는 `OPTIC Broker 주문 등록 workflow`를 기반으로 한다.
- 실제 운영 로직은 재사용하지 않는다.
- 데모는 `mock data + scripted interaction` 중심으로 설계한다.
- 초기 범위는 신규 주문 등록 시나리오 1개다.
- 핵심 메시지는 `입력 -> 정리 -> 요약` 흐름의 전달이다.

## 19. 검토 포인트

다음 리뷰 때 특히 확인이 필요한 항목은 아래와 같다.

- Hero 텍스트와 Preview 중 무엇이 더 먼저 읽혀야 하는가
- Preview 내부 정보 밀도가 랜딩 기준으로 과하지 않은가
- 우측 요약 패널이 충분히 가치 전달을 담당하는가
- 모바일에서 축약 방식이 자연스러운가
- `실제 제품 같음`과 `랜딩 전달력` 사이 균형이 맞는가
