## 1. 배경 및 문제 정의

현재 landing 앱의 `DashboardPreview`(dash-preview Phase 1/2)는 ai-register 화면을 시각적으로 단순화하여 재현했습니다. 핵심 문제는 두 가지입니다.

**신뢰감 한계**: 실제 OPTIC 사용자 또는 시연 경험자가 진짜 화면과의 차이를 인식할 수 있어, 랜딩의 설득력이 약화됩니다.

**조작감 부재**: 버튼을 누르거나 텍스트가 입력되는 "살아있는 느낌"이 없어, Cursor.com의 `demo-window-cursor-ide` 대비 체감 품질이 낮습니다.

Cursor.com demo-window가 동일 방식(실제 구조 복제 + 시각 피드백 레이어)으로 고충실도를 달성한 사례가 검증 레퍼런스로 확보되어 있습니다.

## 2. 제안 내용

### 무엇을 (복제 범위)

`ai-register/page.tsx`의 **main 콘텐츠 영역 전체**를 landing 앱에 시각 모형으로 복제합니다. AiPanel 전체 + OrderForm 전 섹션이 대상입니다.

### 어디까지 (시연 플로우)

시연 플로우는 `INITIAL → AI_INPUT → AI_EXTRACT → AI_APPLY` 4단계로 한정합니다. COMPLETE 단계(등록 확정 연출)는 Phase 2에 유보합니다.

#### AI_APPLY 2단 구조 확장 아이디어 (선택지 제시, 결정 유보)

> 이 항목은 확정된 설계가 아닌 **선택지 제시 단계**입니다. 최종 채택 여부와 방식은 `/plan-draft` 이후 단계에서 결정합니다.

현재 `AI_APPLY` 단계를 **"파트별 적용 → 전체 적용" 2단 구조**로 확장하는 아이디어입니다.

- **파트별 적용**: AI 추출 결과를 카테고리(출발지/도착지/화물/운임) 또는 개별 버튼 단위로 하나씩 적용하는 모습을 시연 → 사용자가 **선택적으로 받을 수 있음**을 전달
- **전체 적용**: 남은 필드를 한 번에 일괄 적용하는 모습을 시연 → **편의성·속도** 강조
- **UX 메시지**: "골라 받을 수도, 한 번에 받을 수도 있다"

구현 방식 3안은 다음과 같으며, 현 단계에서는 모두 열어둡니다.

| 안 | 방식 | 장점 | 단점 |
|----|------|------|------|
| **A — Step 분할** | `INITIAL → AI_INPUT → AI_EXTRACT → AI_APPLY_PARTIAL → AI_APPLY_ALL` (5단계) | 두 모드가 명확히 분리되어 시청자가 이해하기 쉬움 | Phase 1 스펙의 4단계 구조와 어긋남, 재작업 발생 |
| **B — 내부 타임라인 분할 (추천)** | 기존 4단계 유지, `AI_APPLY` 내부 타임라인을 `partial beat → all beat` 2비트로 구성 | Phase 1 스펙 4단계 구조 보존, Step 개수 불변 | 한 Step 안에 2개 연출이 들어가 타이밍 설계 난이도 상승 |
| **C — 모드별 분기** | 자동재생은 안 B(타임라인 분할), 인터랙티브 모드는 안 A(Step 분할) | 시청자 경험과 탐색 경험 각각 최적 | 코드 분기가 늘어나 유지 부담 |

현 단계 추천안: **안 B** — Phase 1 스펙의 4단계 구조를 유지하면서 타임라인 표현만으로 2단 연출을 흡수.

### 어떻게 차별화 (조작감 레이어)

"입력 불가" 시각 모형 기조를 유지하면서, **시각 피드백 전용 인터랙션 레이어 10종**을 추가합니다. state 변경이나 API 호출 없이 조작감만 연출합니다.

MVP 우선 구현 4종:
- **#1** 변동 리듬 타이핑 (AI 입력 텍스트 속도 불규칙 연출)
- **#3** Button Press 물리 (클릭 시 눌림 피드백)
- **#6** Fill-in Caret (폼 필드 커서 깜빡임)
- **#8** 숫자 카운터 롤링 (수치 변경 롤링 전환)

나머지 6종은 Phase 1 스펙 §4에 상세 기술되어 있습니다.

AI_APPLY 2단 구조 채택 시 조작감 레이어와의 연관 (참고):

- **파트별 비트**: 조작감 #3 press + #4 ripple + #6 fill-in caret이 파트마다 순차 발생
- **전체 비트**: 나머지 파트가 짧은 간격으로 연속 발생

> 상세 매핑은 `/plan-draft` 이후 단계에서 다룹니다.
